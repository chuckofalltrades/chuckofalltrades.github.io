Decisions Part 2: Authoring Tool

    May 5, 2024 by Charles Palmer

    I’ve established my goal to create a portfolio site. I’ve decided to use GitHub Pages to host it. Now I need to decide on the tools to build and maintain it.

    The simplest thing to do here would be some very simple HTML with a list of links. If all this is a pure portfolio, that could work.  My plan is for each project to be a GitHub repository, turn on Pages, and bam, you have a link that lets somebody explore that piece of training content. I could just take all those links, stick them on a simple page, and use one of the many templates out there to make it look nice. That would certainly work, but I don’t want this to be just a pure portfolio.

    It’s also important to me to have a blogging aspect because I want to write about these projects and why I do the different things I do. I also want to write about innovations, ideas, tips, tricks, and that sort of thing. So, I need some sort of content management system (CMS).

    When you are exploring GitHub Pages, one of the things that will pop up a lot is Jekyll. Jekyll is an extremely popular static site generator, and that’s what I thought I needed. But when I finally got around to this step and actually looked at Jekyll closely, I realized it was not what I was looking for. Jekyll is really designed with a developer in mind to write content in a very simple way using markdown files. Then Jekyll does the work of converting that into flat HTML files that you then put on your host. I could use that, but that again takes away from my objective of making this site seamless for me so I can focus on the projects.

    I searched for alternatives to Jekyll and found a lot. I looked at Hugo, Decap, and Gatsby. I realized I was going way too far down the wrong path. I was looking at either these static site generators intended to be command-line tools without much of a front end, or stepping back into traditional CMSs that require a backend server to function. These are not what I needed. I backed up and started searching specifically for a flat file generator CMS. After a little bit of digging around, I came across a few options and ultimately decided on one called Publii.

    Publii is an application available for both Windows and Mac. I didn’t really want to go with an application originally because I feel like that locks in my whole process to one computer. But to do what I want to do, I think it’s going to have to be that kind of solution. The more I poked around at Publii, the more I realized it’s pretty much exactly what I wanted. It’s a CMS, like WordPress or Drupal, but it doesn’t work like a traditional CMS running server side. It just publishes out your content into flat files, which then can be pushed to my Git repo. It’s cool because it even has integration to do that directly, it turns out to be a very seamless process

    I’m really liking the tool so far. I’m writing this article in it right now. It’s got all the formatting tools I would expect for a visual editor, but it also has a lot of power under the hood that I can poke around at. What’s nice is because what it’s spitting out are pretty clean flat files, if I want to do something really complicated, there’s nothing stopping me from going into those flat files and editing them directly.

    So, the solution is starting to come together. I’m hosting on GitHub Pages. I’m going to author and manage using Publii. So far, so good.
